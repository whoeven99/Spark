# 对话工作台 - 正确的交互流程

## 改进的设计思路

### 之前的流程（有缺陷）
```
用户对话 → 系统理解意图 → 系统建议选择工具 → 用户操作工具 → 确认执行
（需要多轮对话）
```

### 改进的流程（更直观）
```
用户先选择对象（选择器）→ 用户说任务目标（对话）→ AI 执行
（一步完成，精准高效）
```

---

## 优点分析

### ✅ 更直观
用户思维方式就是："我想对**这些东西**做**这个事**"
- 先指向：点选择器 → 确认对象
- 再说明：自然语言描述任务

### ✅ 更精准
- 对象范围已明确，不需要 AI 猜测
- AI 收到的上下文完整：`{ selectedObjects, userMessage }`
- 减少歧义和多轮对话

### ✅ 更高效
- 一步选择 + 一句对话 = 任务完成
- 不需要"系统反问 → 用户补充"的往返

### ✅ 更可控
- 用户对操作对象的范围有直观的视觉确认
- 不会出现"意外选中了 1000 件商品"的情况

## 对话框的职责变化

### 之前（我的错误设计）
对话框 = 多轮对话不断澄清对象
- 用户："优化库存不足商品"
- 系统："请选择具体商品"
- 用户："那就 CSV 里这些"
- ...（多次往返）

### 改进后（你的正确想法）
对话框 = 在已有对象的基础上，补充任务细节
- 用户先通过选择器选好对象 → 系统显示"已选 168 件"
- 用户在对话框说明任务和参数
- 系统一次理解，直接生成任务

## 对话消息中的上下文信息

当用户发送消息时，系统应该包含：

```typescript
type ChatMessage = {
  id: string;
  role: 'user' | 'assistant';
  content: string;           // 用户的自然语言描述
  timestamp: string;
  
  // 关键：选择的对象上下文
  selectedContext?: {
    objectType: 'products' | 'articles' | ...;
    selectionMethod: 'direct' | 'tags' | 'filter' | 'csv';
    objectCount: number;
    objectIds?: string[];
    filters?: Record<string, unknown>;
    csvData?: { filename: string; content: any[] };
  };
};

// 当 AI 处理消息时，能获得完整上下文：
// {
//   userMessage: "帮我生成专业的商品描述，SEO风格，中等长度",
//   selectedObjects: { count: 168, method: 'tags', filters: {...} },
//   taskType: 'description_generation',  // AI识别
//   parameters: { style: 'seo', length: 'medium' },  // AI提取
// }
```

## 关键优势总结

| 方面 | 之前 | 改进后 |
|------|------|--------|
| 交互步骤 | 多轮对话 | 一次选择 + 一句对话 |
| 用户思维 | 需要适应系统 | 符合用户思维 |
| 对象确认 | 依赖 AI 理解 | 视觉确认 |
| 精确度 | 容易出错 | 100% 精准 |
| 效率 | 低（反复澄清） | 高（一步完成） |
| 可控性 | 低 | 高 |
