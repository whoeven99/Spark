# Spark 任务型工具交互规范

## 1. 文档目标

本文档定义 Spark 中各类 `tools` 页的标准交互结构，用于统一任务型工具的创建、执行、审核与应用流程。

本文档只负责：

- 页面结构
- 任务流程
- 状态定义
- 审核与应用机制
- 任务信息与结果承载方式

本文档不负责：

- 颜色、字体、圆角、阴影等视觉规范
- 组件的视觉风格和设计 token
- 品牌色和品牌字体系

视觉层规范由 [DESIGN.md](file:///Users/cedric/Documents/GitHub/Spark/docs/DESIGN.md) 单独定义。

## 2. 适用范围

本规范适用于 Spark 中所有具备“发起任务 -> 执行任务 -> 产出结果 -> 人工审核 -> 应用结果”链路的工具页，包括但不限于：

- 商品文案生成
- 翻译
- 图片生成或图片处理
- 诊断与批处理类工具
- 未来新增的异步型 AI 工具

## 3. 核心原则

所有任务型工具都必须具备统一的过程意识，而不是一次性操作意识。

统一原则如下：

- 每个工具都必须有独立的“配置页”和“任务页”
- 任何真正消耗时间、积分、token、额度或会写入数据的操作，都必须先创建任务
- 创建任务前必须进行二次确认
- 任务页必须同时承载当前任务和历史任务
- 每个任务无论成功、失败、暂停或中断，都必须有结果产物或过程结果可查看
- 结果进入应用前，必须提供人工审核入口
- 所有任务都应尽量遵循 `创建 -> 进行 -> 审核 -> 应用` 的统一生命周期

## 4. 标准页面结构

每个工具默认由两个一级结构组成：

## 4.1 配置页

配置页用于发起任务。

配置页必须包含：

- 对应功能的参数配置表单
- 明确的主功能按钮
- 必要的辅助说明
- 任务创建前的必要提醒信息

配置页的职责是：

- 让用户明确这次任务要处理什么
- 让用户调整任务参数
- 触发任务创建

配置页不承担：

- 长时间执行过程展示
- 复杂历史任务承载
- 结果审核主承载
- 主操作后的成本确认主承载

## 4.2 任务页

任务页用于承载任务执行、任务结果、任务审核与历史记录。

任务页必须包含两个 tab：

- 当前任务
- 历史任务

历史任务的定义：

- 创建时间超过 `24 小时` 的任务

任务页的职责是：

- 查看任务执行情况
- 查看任务流式进展
- 执行下一步动作
- 进入审核
- 查看历史结果

## 5. 任务创建与二次确认

点击配置页主功能按钮时，不应直接开始最终执行，而应先完成任务创建确认。

标准流程如下：

1. 用户在配置页填写参数
2. 用户点击主功能按钮
3. 系统弹出二次确认弹窗
4. 用户确认后创建任务
5. 系统跳转到任务页，并新增该任务

## 5.1 二次确认弹窗要求

二次确认弹窗必须向用户说明本次任务的基础情况。

至少应包含：

- 任务目标
- 核心参数摘要
- 预计影响范围
- 预估耗时
- 预估积分、token 或额度消耗
- 明确的确认按钮和取消按钮

对于会消耗时间、积分、token、额度或会触发真实写入风险的任务，预估耗时、预估消耗、影响范围等“执行前预估信息”默认只能承载在主操作后的二次确认弹窗中，不应在配置页中作为常驻信息模块重复出现。

配置页可以有轻量提醒文案，但这些提醒文案不能替代二次确认弹窗，也不应形成独立的常驻“执行前预估”卡片、侧栏或面板。

二次确认弹窗的目标是：

- 让用户理解这次任务会做什么
- 让用户理解这次任务要消耗什么
- 避免误触发高成本任务

## 6. 任务页结构

任务页必须以任务列表为核心承载。

每个任务项都应使用统一的任务卡片或任务行结构展示。

每个任务卡片至少需要展示以下信息：

- 任务 ID
- 任务目标
- 任务数量或处理对象数量
- 创建时间
- 更新时间
- 当前状态
- 已完成数量 / 总任务数量
- 百分比进度条
- 流式输出的进展说明
- 结果查看入口
- 审核入口
- 应用入口或后续操作入口

## 6.1 当前任务

当前任务页签用于承载最近 24 小时内、仍在关注窗口内的任务。

其中通常包含：

- 正在创建的任务
- 正在进行的任务
- 待审核的任务
- 待应用的任务
- 最近完成的任务
- 最近失败或暂停但仍需处理的任务

## 6.2 历史任务

历史任务页签用于承载超过 24 小时的任务记录。

历史任务仍应保留：

- 任务基础信息
- 最终状态
- 结果查看入口
- 审核记录或应用记录

历史任务的重点不是继续操作，而是：

- 回溯
- 查询
- 查看结果
- 查看处理记录

## 7. 任务状态与动作

任务状态必须是显式的，不能让用户靠猜测理解任务进展。

工具可以按业务差异扩展状态，但至少应覆盖以下几类：

- 已创建
- 进行中
- 已暂停
- 待审核
- 已审核
- 待应用
- 已完成
- 失败

如果业务中存在“开始”与“创建”分离，也允许保留：

- 未开始
- 已开始

## 7.1 状态驱动动作

不同状态应提供不同的下一步按钮与操作按钮。

例如：

- 未开始 / 已创建：开始
- 进行中：查看进展、暂停
- 已暂停：继续、取消
- 待审核：去审核
- 已审核 / 待应用：去应用
- 已完成：查看结果
- 失败：查看原因、重试

动作设计原则：

- 用户永远能看到当前状态下最合理的下一步
- 不应让所有按钮长期同时出现
- 次要操作应弱于主下一步操作

## 8. 进度展示规范

任务页必须展示任务进度，且进度应同时包含数值进度和语义进度。

## 8.1 数值进度

数值进度至少包括：

- 已完成数 / 总任务数
- 百分比进度条

例如：

- `2 / 4`
- `50%`

## 8.2 语义进度

语义进度用于说明系统当前在做什么。

例如：

- 正在加载数据（2/4）
- 正在翻译
- 正在生成标题
- 正在写入结果
- 正在等待人工审核

语义进度应支持流式更新。

这类信息应尽量：

- 短
- 可扫读
- 与当前阶段一致

## 9. 结果与审核

每个任务不管最终状态如何，都应尽量产出对应的结果、阶段性结果或失败结果说明。

也就是说，任务不应只有“状态”，没有“产物”。

结果承载至少要支持：

- 查看当前结果
- 查看历史结果
- 对比前后内容
- 查看失败信息或中间产物

## 9.1 审核入口

所有任务都必须有结果审核入口。

审核入口可以采用：

- 弹窗
- 抽屉
- 独立详情页

具体形式可以按工具复杂度决定，但必须满足人工可审查。

如果审核或结果查看采用独立详情页承载，则该详情页默认视为任务页内部的二级页面，而不是新的一级导航页面。

这类二级详情页应遵循以下约束：

- 默认不再展示任务页的主导航、主 tab 或顶部统计模块
- 默认不再重复展示工具级主导航
- 页面顶部只保留返回上一页入口，用于回到任务列表
- 详情页主体应专注承载原文、结果、审核、评分、AI 优化与应用动作

## 9.2 审核内容

审核区应支持以下一种或多种能力：

- 前后内容对比
- 原文与结果对比
- 差异高亮
- AI 质量评分
- 人工评分
- 人工编辑修正
- 补充说明与风险提示

审核区的目标不是“再展示一次结果”，而是支持人做最终判断。

## 10. 应用阶段

人工审核确认后，任务应进入应用数据阶段。

应用阶段示例：

- 写入 Shopify
- 覆盖原字段
- 写入 metafield
- 推送到外部系统

应用阶段必须满足：

- 由明确动作触发
- 有单独状态反馈
- 有成功或失败结果

如果应用会产生真实数据写入，就不应在任务执行完成时自动隐式发生。

## 11. 统一生命周期

所有任务型工具都应尽量遵循以下统一生命周期：

1. 创建
2. 进行
3. 审核
4. 应用

如果某些工具存在暂停、失败、重试等分支，也应被视为该生命周期的扩展状态，而不是独立体系。

## 12. 页面级信息层级

为了保持工具页一致性，页面应遵循以下信息层级：

### 配置页

- 页面标题与简要说明
- 参数配置表单
- 主功能按钮
- 任务创建确认入口
- 不承载常驻的执行前预估面板

### 任务页

- 当前任务 / 历史任务切换
- 任务列表
- 单任务进度与状态
- 结果入口
- 审核入口
- 应用入口

## 13. 设计约束

为了防止后续实现偏离，本规范额外约束如下：

- 不允许把异步工具继续做成“一次点击立即写入”的黑盒交互
- 不允许缺失二次确认
- 不允许把执行前预估做成配置页常驻主模块，并与二次确认重复承载
- 不允许只有执行过程、没有审核入口
- 不允许只有审核结果、没有应用阶段
- 不允许历史任务彻底丢失结果查看能力
- 不允许不同工具完全发明不同的任务结构

## 14. 与视觉规范的边界

本文件解决的是：

- 页面模式
- 流程结构
- 状态定义
- 交互阶段
- 审核与应用关系

本文件不解决：

- 卡片长什么样
- badge 用什么颜色
- tab 用什么视觉样式
- 使用何种阴影、圆角、字重和字号

这些内容统一由 [DESIGN.md](file:///Users/cedric/Documents/GitHub/Spark/docs/DESIGN.md) 负责。

## 15. 默认落地模板

未来所有新的任务型工具，在没有特殊说明时，默认采用以下模板：

1. 配置页填写参数
2. 点击主功能按钮
3. 弹出二次确认弹窗
4. 确认后创建任务并跳转任务页
5. 在当前任务中查看执行状态、进度与流式说明
6. 通过审核入口查看结果并进行人工审核
7. 审核通过后触发应用动作
8. 任务完成后进入任务记录，并支持后续查看结果

该模板应作为 Spark 工具页的默认交互骨架。

## 16. Billing Page 交互规范

`Billing Page` 不属于标准任务型工具页，但属于 Spark 的核心商业页，需要单独遵循统一的账户与订阅交互结构。

Billing 页的目标是：

- 让用户快速理解当前账户资源状态
- 让用户理解订阅计划与积分购买的差异
- 让用户完成订阅或加购
- 让用户查询积分历史变动与消耗记录
- 让用户理解不同计划的功能边界与常见问题

## 16.1 页面结构

Billing 页默认由以下几个核心区域组成：

1. 账户概览区
2. 订阅计划区
3. 积分购买区
4. FAQ 区

页面可以按业务需要调整顺序，但默认建议保持以上阅读顺序。

## 16.2 账户概览区

头部必须有独立的账户区域，用于展示当前账户资源状态。

至少需要展示：

- 当前积分余额
- 当前订阅状态
- 当前生效计划
- 计划周期信息
- 下次续费或到期时间
- 积分明细入口

如果账户信息较多，首页默认应只承担：

- 资源状态概览
- 账户详情入口

账务流水、周期归档、订阅明细等信息较多的内容，默认应收拢到 Billing 内部二级页面，而不是在主页面中并列堆叠多个大卡片。

积分明细应支持查看历史增减记录。

积分明细至少需要覆盖：

- 订阅发放
- 积分购买
- 任务消耗
- 手动调整或补偿
- 退款或回退

如果信息较多，可以使用：

- 明细弹窗
- 抽屉
- 独立明细页

但入口必须在头部账户区域中明确可见。

如果采用独立账户详情页承载，则该页面默认应遵循：

- 作为 Billing 页内部二级页面存在
- 顶部只保留返回上一页入口
- 主体按纵向信息流展示订阅、账务流水、周期归档
- 不重复展示套餐选择、购包和 FAQ 等主页面模块

## 16.3 订阅计划区

Billing 页必须支持“订阅计划”和“积分购买”两种商业模式并存。

订阅计划区用于展示不同付费计划。

至少需要支持：

- 月付
- 年付

其中年付必须展示相对月付的折扣信息。

订阅计划区至少应说明：

- 每个计划的价格
- 计费周期
- 核心差异
- 包含的基础额度或权益
- 当前计划标记
- 升级、降级、订阅、续费等入口

如果当前用户已经订阅某计划，也应能清楚看到：

- 当前所在计划
- 可升级到哪些计划
- 切换计划的结果或影响

## 16.4 月付与年付切换

月付与年付应有统一切换控件，不应让用户在多个分散模块中自行比对。

切换后至少要同步更新：

- 价格展示
- 折扣展示
- 计划卡 CTA 文案
- 相关说明文字

折扣信息必须清晰，但不能掩盖计划本身的权益差异。

## 16.5 积分购买区

Billing 页必须提供积分购买区域，作为订阅之外的补充消费能力。

积分购买区至少需要展示：

- 可购买积分包
- 对应价格
- 获得积分数量
- 购买入口

积分购买和订阅计划必须被明确区分，避免用户误以为它们是同一种计费方式。

需要明确说明：

- 订阅适合什么场景
- 积分购买适合什么场景
- 两者是否可以同时存在

## 16.6 计划差异说明

Billing 页需要说明计划差异，但不一定要单独再放一个与套餐选择重复的大模块。

默认要求是：

- 计划差异说明优先并入订阅计划主决策区
- `Choose the plan that fits you` 这一主区域应同时承担价格、周期、核心差异和 CTA
- 不建议在其后再次追加一个语义重复的 `Plan details` 独立大模块

如果确实需要补充说明，也应只补充套餐卡中难以承载的差异信息，而不是重复已有价格、卖点和 CTA。

## 16.7 FAQ 区

Billing 页底部必须有 FAQ 模块，用于解答付费相关的常见问题。

FAQ 至少建议覆盖：

- 订阅如何计费
- 月付和年付有什么区别
- 年付折扣如何计算
- 如何购买积分
- 积分是否会过期
- 任务消耗积分的规则
- 升级或降级计划后会发生什么
- 取消订阅后会发生什么
- 如何查看积分明细与历史消耗

FAQ 应作为页面底部的长期帮助区，不应被折叠到难以发现的位置。

## 16.8 账户与消费透明度

Billing 页必须强调账户状态和消耗透明度。

因此必须满足：

- 用户能看到自己当前拥有什么资源
- 用户能看到资源如何被消耗
- 用户能看到历史增减记录
- 用户能理解计划与积分的关系
- 用户能理解每次付费动作的结果

如果某个操作会改变订阅或积分状态，也应提供明确的确认或结果反馈。

## 16.9 与任务型工具页的区别

Billing 页不是“配置页 + 任务页”结构，也不强制遵循“创建 -> 进行 -> 审核 -> 应用”的任务链路。

Billing 页的默认模式是：

- 账户状态查看
- 商业方案选择
- 购买或订阅决策
- 历史明细查询
- 常见问题解答

因此 Billing 页应被视为 Spark 中的商业与账户页例外模式，而不是任务型工具模板的一部分。

## 16.10 Billing Page 默认信息架构

如果没有特殊说明，Billing 页默认采用以下信息架构：

1. 页面标题与说明
2. 资源状态概览区
3. 账户详情入口
4. 订阅计划区
5. 积分购买区
6. FAQ 区

## 16.10.1 页面标题与说明

页面顶部应先回答用户当前进入该页的目的。

建议包含：

- 页面标题
- 一句简短说明
- 如有必要，展示当前额度不足、订阅待确认或资源告警

## 16.10.2 账户概览区

账户概览区是 Billing 页的第一优先级信息区。

建议拆成两个层次：

- 资源状态概览
- 明细与历史入口

其中资源状态概览应至少包含：

- 当前可用积分
- 已消耗积分
- 订阅状态
- 当前计划
- 周期结束时间或到期时间

明细与历史入口默认应能进入账户详情二级页，承接订阅明细、账务流水和周期归档等更完整的信息。

## 16.10.3 订阅计划区

订阅计划区是 Billing 页的核心决策区。

默认结构建议为：

- 月付 / 年付切换
- 计划卡列表
- 当前计划标识
- 推荐计划标识
- 订阅或变更计划按钮

计划卡中至少需要展示：

- 计划名称
- 价格
- 周期
- 年付折扣
- 核心额度
- 核心卖点
- 当前状态

## 16.10.4 积分购买区

积分购买区应用于补充型消费，不应与订阅计划混在同一组卡片中。

默认结构建议为：

- 积分包选项
- 当前选中积分包摘要
- 购买按钮
- 与订阅的关系说明

如果同时存在订阅和积分购买，页面必须说明：

- 订阅提供什么
- 积分包补充什么
- 哪类用户适合买包而不是升级订阅

## 16.10.5 计划差异说明

计划差异说明默认并入计划选择主区，不再要求单列一个独立的大型详解模块。

至少应覆盖：

- 价格与周期差异
- 核心额度差异
- 当前计划与推荐计划状态
- 用户应选择某个计划的主要理由

## 16.10.6 FAQ 区

FAQ 区默认置于页面底部。

建议采用折叠列表或问答块结构。

每个问题应能够独立阅读，不依赖前文上下文。

## 16.10.7 积分明细承载方式

Billing 页必须提供积分历史增减明细的完整承载方案。

默认可选方案包括：

- 在账户概览区提供“查看明细”入口，打开弹窗
- 在账户概览区提供“查看明细”入口，打开抽屉
- 在 Billing 页中提供独立明细区块
- 跳转独立明细页

积分明细至少应包含：

- 记录时间
- 变动类型
- 增减值
- 变动后余额
- 原因说明
- 关联任务或订单信息

如果是任务消耗，应尽量能看到：

- 任务来源
- 工具名称
- 消耗数量
- 对应任务或结果入口
