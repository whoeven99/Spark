# Spark Tools 交互规范

本文档定义 Spark 内嵌 Shopify App 中各类 **tool 页面** 的通用交互规范，重点约束 **页面层级、任务流程、导航结构、主次操作、状态反馈**。  
视觉表现、色板、卡片样式、间距等请以 `docs/UI_DESIGN.md` 为准；本文只定义“怎么走”“先做什么”“用户下一步去哪里”。

当前以以下页面为首批样板沉淀规则：

- `Product Improve`：`/app/product-improve`
- `Image Studio`：`/app/image-studio`

后续新增任意 tool 页面，应优先复用本文档中的信息架构与流程模式，而不是重新发明一套导航。

---

## 1. 适用范围

| 包含 | 不包含 |
|------|--------|
| `app/routes/app*.tsx` 中与工具页入口相关的导航 | 服务端执行逻辑、模型调用、数据库结构 |
| `app/routes/page/**` 页面级布局与流程组织 | 视觉色值、阴影、圆角细节 |
| `app/routes/component/**` 中承担流程导航职责的组件 | 纯数据计算逻辑 |
| `app/hooks/**` 中直接影响用户流程的前端状态 | Webhook、Prisma、存储客户端 |

冲突处理顺序：

1. 当前用户需求
2. 本文档
3. `docs/UI_DESIGN.md`
4. 现有页面实现

---

## 2. 目标

本规范服务于四个目标：

1. **让用户知道自己现在在哪一步**
2. **让用户知道此刻最应该点击哪个动作**
3. **让用户在任务完成后知道下一步去哪里**
4. **让结果在进入业务系统前有可审查、可确认、可追踪的入口**

判断一个 tool 页面是否符合规范，不看组件多少，而看以下问题是否清楚：

- 页面是否有唯一主任务
- 当前模式是否一眼可见
- 主操作是否唯一且明确
- 结果出现后是否提供下一步动作
- 用户是否可以回退、重试、复用历史

---

## 3. 规范级别

本文档中的规则按以下优先级理解：

| 级别 | 含义 | 处理方式 |
|------|------|----------|
| **必须** | 默认强制执行，除非当前需求明确要求偏离 | 设计评审与开发自检必须检查 |
| **推荐** | 默认采用，若偏离需说明理由 | 允许按业务特性调整 |
| **禁止** | 默认不可使用 | 除非有明确例外说明 |

评审时建议直接用以下问题判断：

- 这个页面的唯一主任务是否清楚
- 页面是否属于明确的流程模型
- 主按钮是否唯一
- 结果区是否承接下一步动作
- 历史区是否真的承担导航价值

---

## 4. 交互原则

### 4.1 单页单主任务

每个 tool 页面只能有一个主任务目标，页面标题、副标题、主按钮必须围绕同一目标。

- `Product Improve`：主任务是“优化商品内容”
- `Image Studio`：主任务是“生成或翻译图片素材”

禁止在同一页面把多个无明显先后关系的高权重任务并列成平级主入口。

### 4.2 先进入任务循环，再进入结果审查与应用

工具页统一遵循以下任务循环：

1. 选择对象
2. 选择任务
3. 配置细化参数
4. 触发主动作
5. 查看预估时间 / 预估积分 / 影响范围
6. 执行任务
7. 展示流式进度
8. 产出结果
9. 审查结果
10. 应用结果
11. 继续加工或返回任务列表 / 配置页

如果某页面不需要“选择对象”，可以直接从“选择任务”或“配置参数”开始；但不得打乱“先输入，后执行，再审查/应用”的顺序。

其中有两种常见承载方式：

- `工作台型`：工具首页承担对象、任务入口、运行中任务、最近结果和历史任务
- `配置型`：默认直接进入配置页，点击主动作后跳转到任务列表，所有任务相关信息集中在任务模块中展示

对异步任务较强、状态较多的工具，推荐把“预估信息”做成点击主动作后的确认弹窗，而不是长期占据配置页空间。

如果任务具备以下任一特征，则推荐完整保留“预估 -> 进度 -> 审查 -> 应用”：

- 存在积分或成本消耗
- 预计执行超过数秒
- 存在批量影响范围
- 会写入 Shopify 或其他业务系统
- 结果需要人工确认后才能应用

如果是批量任务，则要强调进度、影响范围、预计时间和预计积分消耗。

单个任务与批量任务优先共用同一个表单；选中一个对象就是单任务，选中多个对象就是批量任务，除非两者的输入结构存在本质差异。

### 4.3 一个阶段只强调一个主动作

每个流程阶段只保留一个主按钮，其余动作全部降级为辅助动作。

- 主动作：使用 `primary`
- 辅助动作：使用 `secondary`
- 风险动作：放到结果区、弹窗或危险区，不与主按钮并排竞争

### 4.4 结果不是终点，必须带下一步

所有 tool 的结果区至少要提供以下动作中的一个：

- 保存
- 复制
- 重新生成 / 重试
- 切换模式
- 查看历史
- 返回上一步修改输入

### 4.5 历史是二级导航，不是页面垃圾桶

历史记录不是简单堆列表，而是工具页的“复用入口”和“回看入口”。

历史区承担以下职责：

- 快速回到最近任务
- 横向比较不同结果
- 复用之前的输入或结果
- 删除无效记录

## 5. 页面层级模型

大多数 task-driven tool 推荐按五层组织：

### 5.1 L1 应用级导航

由应用壳提供，负责在大能力之间切换。

示例：

- AI Assistant
- Diagnosis
- Translation
- Product Improve
- Image Studio
- Billing

该层由 `app/routes/app.tsx` 维护，属于全局导航，不承载单个工具的局部流程。

### 5.2 L2 工具首页 / 配置页

L2 层不一定总是“工作台首页”，也可以是“默认配置页”。

示例：

- `/app/product-improve`
- `/app/image-studio`

要求：

- L2 页必须有明确标题
- L2 页必须有一句简短副标题说明“这个工具帮助你完成什么”
- 若为工作台型，L2 页必须能承接任务执行后的回流状态
- 若为配置型，L2 页必须优先承载真实表单，不应被说明性卡片占满
- 对象选择优先使用资源选择器、选择器入口或搜索选择组件，不推荐手写静态对象说明区

### 5.3 L3 任务页

任务页负责“配置参数 -> 查看预估 -> 执行任务”，是用户真正发起任务的地方。对于配置型工具，任务列表也可以作为 L3 的主要承接层。

示例：

- `/app/product-improve/task/...`
- `/app/image-studio/task/...`

要求：

- 任务页必须说明当前在执行什么任务
- 任务页必须展示参数配置区域
- 任务页必须在执行前展示预估信息（若任务支持，可用弹窗承接）
- 任务页必须在执行后进入结果、审查或任务列表

### 5.4 L4 工具内模式导航

当同一工具下存在多个模式时，必须在页面内提供显式模式导航。

适用场景：

- 同一个工具下有多个平级能力
- 用户切换模式后，输入区、结果区、历史口径会变化

推荐形式：

- `Tab`
- `Segmented Control`

当前样板：

- `Image Studio` 使用 `生成 / 翻译` 两个模式

不满足以下条件时，不要强行增加模式层：

- 两个动作其实是同一主流程中的步骤
- 切换后用户收益不明显
- 会让主任务变得模糊

### 5.5 L5 结果、审查与应用层

任务执行后，结果默认不能直接等于应用，推荐拆成三层：

1. `Result Asset`：工具产出的结果
2. `Review Gate`：审查、确认、筛选、编辑
3. `Apply Action`：写入 Shopify 或触发下游动作

当结果数量较多、结构较复杂或需要批量审查时，可以独立出结果落地页。

## 6. 页面规范矩阵

| 项目 | 必须 | 推荐 | 禁止 |
|------|------|------|------|
| 页面身份 | 有明确标题与副标题 | 在标题区提供状态 badge | 无标题直接进入表单 |
| 工具首页 | 可发起任务并承接结果回流 | 展示运行中任务和最近结果 | 只有介绍，没有任务入口 |
| 任务页 | 承载配置与执行 | 展示预估时间/积分/影响范围 | 把复杂配置塞回首页 |
| 主任务 | 只有一个主任务 | 用一句话说明结果导向 | 一个页面同时承载多个并列主任务 |
| 流程模型 | 明确属于“任务循环”及其子模型 | 在评审稿写明流程类型 | 混合多套心智模型且无分层 |
| 主按钮 | 一个阶段只有一个主按钮 | 用动作动词命名 | 多个主按钮并列竞争 |
| 结果区 | 提供下一步动作 | 提供回退或重试 | 只展示结果，不给后续动作 |
| 审查层 | 写入前有审查或确认入口 | 支持 diff、筛选、跳过 | 结果直接自动写入业务系统 |
| 历史区 | 能选中、识别状态 | 支持删除和复用 | 只堆数据、不提供回看入口 |
| URL | 工具页有稳定主路径 | 模式切换用查询参数 | 同一工具拆出大量平级碎路由 |
| 错误反馈 | 贴近出错区块展示 | 指明用户下一步如何修复 | 仅靠 toast 表达全部错误 |

## 7. 标准页面骨架

task-driven tools 默认包含两类常见承载方式：

1. `Tool Home`：任务中枢
2. `Config + Task List`：配置页负责发起任务，任务列表负责承接全部任务状态

复杂结果可再增加：

3. `Result Landing`：结果落地页

### 7.1 工作台型（Tool Home）

```text
s-page
└── pageContent
    ├── 页面标题区
    ├── 模式导航区（可选）
    ├── 对象选择区
    ├── 任务入口区
    ├── 运行中任务区
    ├── 最近结果区
    ├── 历史任务区
    ├── 二次加工区
    └── 页脚说明区
```

### 7.2 配置型（Config + Task List）

```text
s-page
└── pageContent
    ├── 页面标题区
    ├── Tab A 配置页
    │   ├── 商品 / 对象选择器入口
    │   ├── 参数表单
    │   ├── 主动作
    │   └── 预估弹窗（点击主动作后出现）
    ├── Tab B 任务列表
    │   ├── 执行中任务卡
    │   ├── 待审查任务卡
    │   ├── 已应用任务卡
    │   └── 历史任务卡
    └── 页脚说明区
```

适用场景：

- 用户更关注“发起任务”和“查看任务状态”
- 任务执行具有明显的异步特征
- 审查、应用、继续加工都可以收敛成任务状态
- 配置页不需要承担大量说明性内容

### 7.3 页面标题区

要求包含：

- 页面标题
- 页面副标题
- 可选状态 badge

标题回答“这是什么工具”，副标题回答“它帮你完成什么结果”。

### 7.4 模式导航区

仅在存在多个平级模式时出现，放在标题区之后、主工作区之前。

要求：

- 当前模式必须高亮
- 切换后 URL 或内部状态要可恢复
- 切换后页面标题可以不变，但区块标题必须同步变化

### 7.5 任务页（Task Page）

优先采用以下骨架：

```text
s-page
└── pageContent
    ├── 任务标题区
    ├── 当前对象摘要
    ├── 参数配置区
    ├── 预估信息区
    ├── 主执行区
    ├── 流式进度区
    ├── 结果摘要区
    └── 返回首页 / 继续审查入口
```

### 7.6 主工作区

任务页的主工作区必须回答以下问题：

- 我要输入什么
- 我现在可以执行什么
- 结果会出现在哪里
- 审查会在哪里发生
- 应用动作会在哪里发生

推荐布局：

- 输入在左 / 上
- 结果在右 / 下
- 执行动作紧贴输入区

### 7.7 历史区

历史区默认放在主工作区之后，除非该工具历史价值极高，才允许提升到同层主区。

历史区应支持：

- 选中当前记录
- 删除记录
- 识别记录类型
- 识别记录状态

### 7.8 页脚说明区

页脚只放以下内容：

- 规则说明
- 风险提醒
- 数据来源说明
- 能力边界说明

禁止在页脚放新的主操作。

## 8. 业务流程导航规范

### 8.1 流程分型

工具页流程优先先判断是否属于“任务循环流”，再判断其内部采用哪种子模型。

| 类型 | 特征 | 适用 |
|------|------|------|
| 任务循环流 | 工作台首页或配置页 -> 任务承接层 -> 结果/审查 -> 返回首页或任务列表 | 大多数 tool |
| 串行任务流 | 单个任务页内部存在明确的先后阶段 | `Product Improve` |
| 工作台模式流 | 工具首页内存在多个可切换 mode | `Image Studio` |

规则：

- 工具级结构优先使用 `任务循环流`
- 单个任务页内部如有明显先后阶段，可再使用 `串行任务流`
- 工具首页如有多个平级能力，可再使用 `工作台模式流`
- 不要在同一视觉层级同时混用多套心智模型

### 8.2 任务循环流

适用于大多数需要历史、进度、结果回流和二次加工的 tool。

标准结构：

1. 在工作台首页或配置页选择对象与任务
2. 在任务页或配置区完成参数配置
3. 查看预估时间、积分、影响范围
4. 执行任务并展示流式进度
5. 产出结果
6. 进入审查层
7. 应用到业务系统或返回任务列表 / 首页
8. 结果回流到首页、任务列表或历史区，继续加工或沉淀为历史

规范要求：

- 工作台首页或任务列表必须承接任务回流状态
- 任务页、配置区或任务卡必须承接执行与进度
- 结果与应用必须分离
- 任务结束后必须给出“回首页 / 回任务列表 / 继续审查 / 继续加工”路径

### 8.3 串行任务流

适用于 `Product Improve` 一类页面。

标准步骤：

1. 选商品
2. 配参数
3. 生成内容
4. 审核编辑
5. 保存回写

规范要求：

- 顶部或区块内应能看出当前任务阶段
- 主按钮文案体现当前动作
- 次要分析动作不与主按钮平权
- 保存前必须有明确确认机制

### 8.4 工作台模式流

适用于 `Image Studio` 一类页面。

标准结构：

1. 选择模式
2. 录入模式所需输入
3. 执行
4. 查看结果
5. 从历史回看或复用

规范要求：

- 模式切换必须稳定
- 切换模式后历史和结果上下文要同步
- 不同模式的区块标题和空状态文案必须不同

## 9. 主次操作规范

### 9.1 主按钮

每个主要流程区只保留一个主按钮，文案必须是动作动词，而不是抽象名词。

推荐：

- 生成优化内容
- 开始批量优化
- 生成图片
- 开始翻译
- 保存到 Shopify

不推荐：

- 提交
- 开始
- 确认
- 下一步

### 9.2 次级动作

以下动作默认降级为次级动作：

- 清空
- 重置
- 复制
- 查看详情
- 打开原图
- 删除历史

若某动作不可逆或成本高，不得伪装成普通次级动作。

### 9.3 危险动作

危险动作应满足以下约束：

- 不与主按钮紧邻
- 有明确的危险语义
- 需要确认或二次确认

示例：

- 删除历史记录
- 覆盖保存
- 取消已生效操作

### 9.4 操作优先级

同一操作区建议按以下顺序摆放：

1. 主动作
2. 紧邻主动作的辅助动作
3. 低频辅助动作
4. 危险动作

如果页面存在超过 4 个同层动作，必须重新判断是否需要：

- 折叠高级操作
- 将部分动作移动到结果区
- 将部分动作移动到历史项内
- 将部分动作改成弹窗内确认

### 9.5 审查与应用动作

对会写入 Shopify 或其他业务系统的任务，建议显式区分以下动作：

- 结果动作：查看、编辑、复制、比较、筛选
- 审查动作：通过、拒绝、跳过、标记待处理
- 应用动作：写入 Shopify、批量应用、导出后应用

规则：

- `Apply` 不应与普通辅助动作混在同一层
- 审查通过前，不应默认直接写入业务系统
- 批量任务应允许“部分通过、部分跳过、部分失败”

## 10. 状态反馈规范

每个工具页必须完整覆盖以下状态中的适用项：

### 10.1 初始空状态

要求：

- 说明当前工具能做什么
- 提示第一步该做什么
- 不要只写“暂无数据”

### 10.2 执行中状态

要求：

- 主按钮进入 loading 或 disabled
- 结果区显示处理中说明
- 如果是异步任务，明确告诉用户正在生成、轮询或等待返回

### 10.3 成功状态

要求：

- 结果可见
- 下一步动作可见
- 若支持 toast，可补成功反馈，但不能只依赖 toast

### 10.4 错误状态

要求：

- 错误贴近出错区块展示
- 用户知道是否可以重试
- 如果是输入错误，要直接指向需要改的字段

### 10.5 历史恢复状态

当用户从历史选中旧记录时，页面必须清楚区分：

- 当前是新任务
- 当前是在查看历史任务

推荐方式：

- 高亮当前选中记录
- 结果区保留任务来源提示
- 模式自动切换到对应能力

### 10.6 审查中状态

要求：

- 明确告诉用户“结果已生成，但尚未应用”
- 标明当前可审查的对象范围
- 支持通过、拒绝、跳过、继续编辑中的一种或多种

### 10.7 应用中状态

要求：

- 明确告诉用户正在写入哪个业务系统
- 若为批量应用，需展示成功数、失败数、剩余数
- 应用过程中不应让用户误以为结果已经全部落库

### 10.8 已应用状态

要求：

- 展示真实耗时、真实积分消耗、应用结果
- 区分“结果生成成功”和“应用成功”
- 若存在失败项，应有单独的失败说明与后续处理入口

---

## 11. 导航与 URL 规范

### 11.1 工具级路径

每个工具应有稳定、可分享的主路径：

- `/app/product-improve`
- `/app/image-studio`

### 11.2 模式级参数

工具内模式切换优先使用查询参数表达，而不是扩展多个平级路由。

示例：

- `/app/image-studio?tab=generate`
- `/app/image-studio?tab=translate`

适用前提：

- 页面壳一致
- 模式间只变工作区内容，不变整体页面定位

### 11.3 兼容旧入口

旧链接允许保留，但应统一重定向到标准工具入口。

示例：

- `/app/generate-image` → `/app/image-studio?tab=generate`

### 11.4 任务页与结果页路径

推荐按以下语义组织：

- 工具首页：`/app/<tool>`
- 任务页：`/app/<tool>/task/<taskType>` 或等价语义路径
- 结果页：`/app/<tool>/result/<taskId>`（仅在批量或复杂结果需要独立承接时）

原则：

- 首页负责聚合与回流
- 任务页负责配置与执行
- 结果页只在确有必要时独立存在

## 12. 样板落地

### 12.1 Product Improve

页面类型：**任务循环流 + 配置型**

建议层级：

1. 默认进入配置页
2. 通过商品选择器入口选择单个或多个商品
3. 在同一表单中配置目标语言、风格、范围等参数
4. 点击 `生成内容` 后弹出预估信息
5. 确认后自动进入任务列表
6. 所有执行中、待审查、已应用状态都在任务列表中承接
7. `质量评分` 作为辅助动作存在于配置页或任务卡中

规则：

- `生成内容` 是唯一主动作
- `质量评分` 是辅助分析动作，不与 `生成内容` 平权
- 配置页不应堆叠任务摘要、任务规则、对象介绍等说明性卡片
- 商品选择优先采用 Shopify 资源选择器或等价入口
- 保存动作必须发生在任务卡的审查状态之后
- 结果应先进入 `Review Gate`，再进入 Shopify 写入
- 单任务与批量任务应共用同一表单，通过已选对象数量自动区分

页面流程示意：

```text
Product Improve
└── 页面标题区
    ├── 页面标题：商品优化
    ├── 页面副标题：配置参数、发起任务，并在任务列表中完成审查与写入
    └── 状态提示：计费 / 权限 / 语言来源
└── Tab A 配置页
    ├── 商品选择器入口
    ├── 参数表单
    ├── 质量评分（辅助动作）
    └── 点击主按钮后弹出预估信息
└── Tab B 任务列表
    ├── 执行中任务卡
    │   └── 流式进度 / 执行日志
    ├── 待审查任务卡
    │   └── 审查结果 / 写入 Shopify
    ├── 已应用任务卡
    │   └── 真实耗时 / 真实积分 / 继续加工
    └── 历史任务卡
└── 页脚说明区
```

评审重点：

- 配置页是否只保留真实表单，而没有混入任务说明卡
- 商品选择是否通过标准选择器入口承接，而不是手写对象介绍
- 预估信息是否发生在点击主动作之后，而不是常驻在表单旁
- 任务列表是否真正承接了执行中、待审查、已应用三种状态
- 是否清楚区分“结果生成成功”和“已写入 Shopify”

### 12.2 Image Studio

页面类型：**任务循环流 + 工作台模式流**

建议层级：

1. 工具首页：图片工作台
2. 模式导航：
   - 图片生成
   - 图片翻译
3. 任务入口区：
   - 单图任务
   - 批量任务
4. 进入任务页：
   - 配置参数
   - 查看预估时间与积分
   - 执行任务
5. 返回首页：
   - 查看流式进度
   - 查看最近结果
   - 对结果继续加工
6. 多结果时：
   - 进入结果落地页做集中展示和二次处理

规则：

- `tab` 是模式切换，不是步骤切换
- 历史区属于页面级共享资产
- 切换历史记录时允许自动切换到对应模式
- 如结果后需要发布或写回，也应先进入审查层

页面流程示意：

```text
Image Studio
└── 页面标题区
    ├── 页面标题：图片工作台
    └── 页面副标题：生成或翻译店铺图片素材
└── 工具首页
    ├── 模式导航区
    │   ├── Tab A 图片生成
    │   └── Tab B 图片翻译
    ├── 任务入口区
    ├── 运行中任务区
    ├── 最近结果区
    └── 统一历史区
└── 任务页
    ├── 参数配置区
    ├── 预估信息区
    ├── 执行动作
    └── 流式进度区
└── 结果层
    ├── 单结果直接展示
    └── 多结果进入落地页
└── 页脚说明区
```

评审重点：

- `tab` 是否承担了模式切换，而非步骤切换
- 模式切换后标题、结果、历史是否都同步
- 历史是否真正支持回看与复用，而不只是列表展示
- 单结果与多结果是否采用了不同承接方式

## 13. 新工具接入清单

### 13.1 接入前问答

新增 tool 页面时，必须先回答以下问题，再开始画 UI：

1. 这个页面的唯一主任务是什么？
2. 它属于任务循环流、串行任务流还是工作台模式流？

6. 结果出来后，用户最常见的下一步是什么？
7. 它是否需要 `Review Gate`？
8. 历史对这个工具是“必需”还是“可选”？
9. 模式切换是否需要进入 URL？
10. 单任务与批量任务能否共用同一表单？
11. 预估信息应常驻展示，还是点击主动作后用弹窗确认？

### 13.3 评审清单

#### 必须

- 页面有明确标题与副标题
- L2 页能清楚承担“工作台首页”或“默认配置页”的角色
- 页面只表达一个主任务
- 页面明确属于一种流程模型
- 主流程中同层只有一个主按钮
- 结果区提供清晰下一步动作
- 写入前存在审查或确认入口
- 错误状态不只依赖 toast
- 历史区如存在，必须承担回看或复用

#### 推荐

- 模式切换进入 URL
- 页面支持从历史恢复上下文
- 页面保留页脚边界说明
- 结果区支持重试或回退
- 对象信息在任务执行后持续可见
- 任务页展示预估时间、预估积分和影响范围
- 首页展示真实耗时、真实积分和最终状态
- 对象选择优先采用标准资源选择器入口
- 单任务与批量任务优先共用同一表单
- 对异步任务优先用主动作后的预估弹窗做确认

#### 禁止

- 一个页面同时承载多个并列主任务
- 多个主按钮抢占同一视觉层级
- 把模式切换做成看似步骤导航
- 把结果区做成纯展示，不给下一步
- 把历史区当作普通表格堆叠
- 结果未经审查直接写入业务系统
- 在配置页堆叠任务摘要、任务规则、对象介绍等说明性卡片

## 14. 跨工具统一规则

以下规则默认适用于后续所有 tools，除非该工具有明确业务例外：

1. 工具页必须有稳定主路径。
2. L2 页必须有可独立理解的页面标题与副标题。
3. 大多数 tool 优先采用“工作台首页”或“默认配置页”二选一，而不是两者同时堆叠。
4. 对状态较多的异步工具，优先采用“配置页 + 任务列表”的双 tab 承载。
5. 同一工具内部若存在多个平级能力，优先做模式导航，而不是分裂成多个孤立页面。
6. 任务页或配置页的主工作区必须围绕“输入 -> 执行 -> 结果/审查”组织。
7. 历史区默认是二级导航资产，不是附属表格。
8. 写入型任务必须区分“结果生成成功”和“应用成功”。
9. 配置页优先保留真实表单，不承载大段任务规则说明。
10. 页脚说明区只负责边界、风险、规则，不承载新主动作。
11. 若某工具暂时没有历史区，应在接入评审时明确说明原因。

## 16. 术语与命名规范

为避免产品、设计、前端对同一概念使用不同名称，后续工具页评审默认使用以下术语。

### 16.1 核心术语

| 术语 | 定义 | 说明 |
|------|------|------|
| Tool | 面向用户的独立能力单元 | 通常有独立页面或明确入口 |
| Tool Home | 某个 tool 的首页与任务中枢 | 如 `Product Improve`、`Image Studio` |
| Config Page | 默认进入的配置页 | 用于填写参数、发起任务 |
| Task Page | 用户配置并执行任务的二级页面 | 承载参数、预估、执行、进度 |
| Task List | 统一展示任务状态的列表页或任务区 | 承接执行中、待审查、已应用、历史 |
| Mode | 同一 tool 内的平级工作模式 | 如图片生成、图片翻译 |
| Step | 串行流程中的先后阶段 | 如选商品、生成、保存 |
| Input Panel | 用户录入或选择参数的区域 | 表单、选择器、上传区 |
| Result Panel | 展示输出结果与后续动作的区域 | 预览、复制、筛选、重试 |
| Result Asset | 工具产出的结果资产 | 不等于已经写入业务系统 |
| Review Gate | 结果进入应用前的审查层 | 可编辑、筛选、通过、拒绝 |
| Apply Action | 把结果真正写入业务系统的动作 | 如写入 Shopify、批量发布 |
| History Panel | 承担回看、复用、恢复上下文的区域 | 不是简单日志列表 |
| Primary Action | 当前阶段最重要的唯一动作 | 一般为 `primary` 按钮 |
| Secondary Action | 对主流程有帮助但不主导的动作 | 如重置、复制、查看详情 |
| Dangerous Action | 不可逆或高风险动作 | 删除、覆盖、取消生效操作 |
| Empty State | 初始或无结果时的占位状态 | 需告诉用户第一步做什么 |
| Recovery | 从历史或失败状态恢复流程的能力 | 包括回退、重试、恢复上下文 |

### 16.2 命名使用规则

#### 必须

- 页面级能力使用 `Tool` 或具体工具名表述，如 `Product Improve`、`Image Studio`
- 工具首页统一称为 `Tool Home`
- 默认进入的表单页统一称为 `Config Page`
- 二级配置执行页统一称为 `Task Page`
- 统一展示任务状态的区域统一称为 `Task List`
- 同一工具下的平级能力统一称为 `Mode`
- 串行先后关系统一称为 `Step`
- 当前阶段唯一重点按钮统一称为 `Primary Action`
- 写入前的确认区域统一称为 `Review Gate`

#### 推荐

- 页面内分区统一使用 `Input / Result / History / Footer` 语义
- 涉及写入时统一使用 `Result / Review / Apply` 语义链路
- 在设计稿与评审文档中直接标注 `任务循环流` 及其子模型
- 对外用户文案可以更自然，但内部讨论应坚持统一术语

#### 禁止

- 把 `Mode` 和 `Step` 混用
- 把 `Result` 和 `Apply` 视为同一个动作
- 把 `History` 叫成“列表”但实际上承担恢复上下文
- 把多个同级工具能力随意称为 tab，而不说明它究竟是模式还是步骤

### 16.3 推荐命名模式

| 场景 | 推荐命名 | 不推荐命名 |
|------|----------|------------|
| 工具首页标题 | 商品优化工作台、图片工作台 | 工具介绍页、能力中心 |
| 模式名 | 图片生成、图片翻译 | 功能一、功能二 |
| 主按钮 | 生成优化内容、保存到 Shopify | 提交、开始、确认 |
| 历史区标题 | 最近任务、历史记录 | 数据列表 |
| 结果区标题 | 生成结果、翻译结果 | 输出内容 |
| 审查区标题 | 待审查结果、确认应用 | 最终步骤 |

## 17. 异常流与导流规范

工具页不仅要定义“正常情况下用户怎么走”，还要定义“走不通时把用户带去哪里”。

### 17.1 异常流目标

异常流设计必须回答三个问题：

1. 用户为什么现在不能继续
2. 用户应该修改什么或等待什么
3. 修复后应该回到哪一步

### 17.2 异常类型

| 类型 | 示例 | 默认处理方式 |
|------|------|--------------|
| 输入缺失 | 未选商品、未填描述、未选语言 | 贴近字段提示 + 禁止执行 |
| 权限不足 | 缺 scopes、缺 Shopify 权限 | 顶部告警 + 导向授权或说明页 |
| 计费受限 | 余额不足、未订阅 | 顶部告警 + 导向计费页（仅对启用计费的 tool） |
| 异步处理中 | 任务生成中、轮询中 | 结果区明确处理中状态 |
| 异步失败 | 超时、远端服务失败 | 结果区错误 + 可重试 |
| 空结果 | 返回成功但无有效内容 | 结果区空态 + 修改输入建议 |
| 历史失效 | 历史项已删除、资源过期 | 提示历史失效 + 返回当前模式初始态 |
| 审查未通过 | 结果被拒绝、字段需重做 | 返回结果编辑或重新发起任务 |
| 应用失败 | 写入 Shopify 失败、批量部分失败 | 展示失败项 + 支持重试或跳过 |

### 17.3 导流规则

#### 必须

- 输入错误导回当前输入区，不允许直接把错误埋在 toast 中
- 权限类问题必须指向“如何获得权限”或“当前无法继续”的明确说明
- 对启用计费的 tool，计费类问题必须提供进入 `/app/billing` 的入口
- 异步失败必须提供重试或重新发起任务的方式
- 历史失效必须允许用户回到当前工具的可执行初始态
- 审查未通过必须允许用户回到结果编辑或重新生成
- 应用失败必须明确区分失败原因、失败对象和后续处理方式

#### 推荐

- 顶部 `banner` 负责全局限制类问题，如计费、权限、账号状态
- 区块内错误框负责局部错误，如字段缺失、某次请求失败
- 结果区保留“返回修改输入”的路径，避免用户在失败后迷失

#### 禁止

- 只显示“失败了，请稍后重试”而不告诉用户下一步
- 只弹一次 toast 就结束整个异常处理
- 出现异常后让用户丢失当前上下文

### 17.4 异常流处理矩阵

| 异常 | 出现位置 | 展示方式 | 下一步 |
|------|----------|----------|--------|
| 未完成输入 | 输入区 | 字段错误或表单错误框 | 回到字段补全 |
| 权限不足 | 页顶或主工作区 | `banner` + 说明文案 | 去授权 / 知道暂不可用 |
| 余额不足 | 页顶 | `banner` + 跳转计费页 | 去 `/app/billing` |
| 生成失败 | 结果区 | 错误框 + 重试按钮 | 重试或修改输入 |
| 返回空结果 | 结果区 | 空态说明 + 调整建议 | 修改输入后再执行 |
| 历史项无效 | 历史区或结果区 | 提示 + 清理当前选中态 | 返回模式初始态 |
| 审查未通过 | 审查区 | 拒绝原因 + 返回编辑入口 | 回到结果编辑或重跑 |
| 应用失败 | 审查区或结果页 | 失败说明 + 重试/跳过 | 处理失败项或返回首页 |

### 17.5 样板要求

#### Product Improve

- 商品未选择时，错误必须回到商品选择区
- 语言未选择时，错误必须回到语言选择区
- 保存失败时，结果编辑区必须保留用户已编辑内容
- 余额不足时，应直接导向计费页，而不是只提示失败
- 审查后写入 Shopify 失败时，应明确告知失败字段或失败商品

#### Image Studio

- 异步生成中，结果区必须持续说明当前正在处理中
- 图片翻译失败后，应保留已选图片或当前上下文
- 历史资源失效时，应提示用户重新生成或重新上传
- 若后续涉及发布或落库，应区分“素材已生成”和“素材已应用”

## 18. Tool 立项与分级规则

并不是所有能力都应该成为独立 tool。新增能力前，先判断它属于以下哪一类。

### 18.1 能力分级

| 级别 | 定义 | 典型形态 |
|------|------|----------|
| L1 独立 Tool | 有独立主任务、独立流程、独立结果区 | 独立页面 |
| L2 Tool Mode | 属于现有 tool 的一个平级模式 | tab / segmented |
| L3 Tool Subflow | 属于某个 tool 主流程中的一个步骤或辅助区 | 页面内区块、弹窗、侧栏 |
| L4 Review / Apply Surface | 用于审查结果、确认写入的承接层 | 审查页、审查抽屉、批量应用页 |
| L5 Chat Skill | 更适合在聊天中即时触发，不需要独立页面 | 聊天卡片、对话动作 |

### 18.2 何时做成独立 Tool

#### 必须满足至少 4 条

- 该能力有独立主任务
- 该能力需要完整输入区、结果区和异常流
- 用户会多次重复使用
- 该能力有可持续沉淀的历史或资产
- 该能力与现有工具模式差异较大
- 该能力需要独立导航曝光

#### 典型例子

- `Product Improve`
- `Image Studio`
- `Translation`
- `Billing`

### 18.3 何时做成 Tool Mode

满足以下条件时，更适合成为现有 tool 的 mode：

- 与现有 tool 共享同一页面身份
- 共享历史区或结果资产
- 只是输入方式、输出类型不同
- 用户在这些能力之间会高频切换

典型例子：

- `Image Studio` 下的图片生成 / 图片翻译

### 18.4 何时做成 Tool Subflow

满足以下条件时，更适合成为页面内子流程，而不是独立 tool：

- 它是某个主任务的辅助判断
- 它不需要独立导航曝光
- 它没有独立结果闭环
- 它更适合作为主流程中的一步

典型例子：

- `Product Improve` 中的商品质量评分
- 保存前确认弹窗
- 高级参数折叠区

### 18.5 何时做成 Review / Apply Surface

满足以下条件时，建议把结果后的“审查与应用”单独承接，而不是直接在结果区一键写入：

- 会写入 Shopify 或其他业务系统
- 有覆盖风险或批量风险
- 需要 diff、筛选、跳过、部分通过
- 需要区分“生成成功”和“应用成功”

典型例子：

- 批量商品文案写回 Shopify
- 批量翻译结果审核后发布
- 多图片结果筛选后再应用到商品

### 18.6 何时留在 Chat Skill

满足以下条件时，不建议单独做页面：

- 低频使用
- 结果短平快
- 没有复杂输入
- 没有强历史复用需求
- 用户更适合在对话里顺手触发

典型例子：

- 快速问答
- 简单文本生成
- 一次性建议卡片

### 18.7 立项判断清单

新增能力前，评审人应先回答：

1. 这是不是独立主任务？
2. 它是否需要独立导航入口？
3. 它是否有独立结果区与后续动作？
4. 它是否需要历史沉淀？
5. 它是否真的值得用户离开聊天流？
6. 它是否可以并入已有 tool 的 mode？
7. 它是否需要独立的审查 / 应用承接层？
8. 它是否只是现有流程中的一个辅助步骤？

如果以上问题回答不清楚，默认不要新建独立 tool。

## 19. Agent 执行清单

1. 确认该需求是在改“流程”还是改“视觉”。
2. 若改流程，先读本文档，再读 `docs/UI_DESIGN.md`。
3. 先判断页面是否属于“任务循环流”，再判断其子模型。
4. 检查页面是否存在多个互相竞争的主按钮。
5. 检查结果区是否提供清晰的下一步动作。
6. 检查历史区是否承担复用与回看，而不是只做展示。
7. 在说明中标注本次改动遵循了哪种流程模型。
