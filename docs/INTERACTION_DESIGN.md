# Spark 交互规范

## 1. 文档目标

本文档定义 Spark 的交互层级、页面类型与页面级规则，用于统一全局导航秩序，并为不同类型页面建立稳定的默认交互骨架。

本文档只负责：

- 页面层级
- 导航与返回规则
- 弹窗、抽屉、子页面的使用边界
- 页面类型的默认交互骨架
- 工具页的任务流程、状态定义、审核与应用机制
- 页面级规则如何在全局规则与类型规则之下补充

本文档不负责：

- 颜色、字体、圆角、阴影等视觉规范
- 组件的视觉风格和设计 token
- 品牌色和品牌字体系

视觉层规范由 [DESIGN.md](file:///Users/cedric/Documents/GitHub/Spark/docs/DESIGN.md) 单独定义。

## 2. 规范层级与适用范围

Spark 的交互规范分为三个层级，优先级从高到低如下：

### 2.1 全局规则

全局规则适用于整个应用，不因页面类型变化而失效。

全局规则负责定义：

- 应用级导航、页面级模式切换、详情返回等层级关系
- 弹窗、抽屉、子页面的决策边界
- 主次动作层级
- 多级 tab 的上限与替代方案
- 共享交互基座的优先级

### 2.2 类型规则

类型规则适用于某一类页面，用于定义该类页面的默认工作模式与默认信息结构。

Spark 当前默认区分以下页面类型：

- 工具页
- 对话页
- 商业页
- 工作台页

### 2.3 页面规则

页面规则仅适用于某个具体页面或某个极小页面族。

页面规则只能：

- 细化该页面的业务特征
- 补充该页面的对象模型
- 说明该页面相对所属类型规则的局部特殊点

页面规则不得：

- 推翻全局规则
- 绕开所属页面类型的默认骨架
- 把临时实现升格为全局模式

### 2.4 页面类型定义

#### 工具页

适用于具备“发起任务 -> 执行任务 -> 产出结果 -> 人工审核 -> 应用结果”链路的页面，包括但不限于：

- 商品文案生成
- 翻译
- 图片生成或图片处理
- 诊断与批处理类工具
- 未来新增的异步型 AI 工具

#### 对话页

适用于以连续对话流、上下文注入、嵌入式卡片与实时反馈为主体的页面，例如 AI Assistant。对话页可以嵌入工具卡、任务卡、提案卡，但页面本身不应被误判为工具页。

#### 商业页

适用于账户、订阅、积分购买、账务明细、FAQ 等商业决策页面，例如 Billing。商业页可以使用更强的价格层级与推荐表达，但这些表达不得反向污染工具页。

#### 工作台页

适用于多对象管理、筛选、上下文选择、批量操作、跨模块协作类页面，例如 Workspace。工作台页强调对象管理与上下文组织，不默认套用工具页的“配置页 / 任务页”结构。

### 2.5 适用范围

本文件对全局规则适用于整个 Spark。

其中：

- 工具页规则是本文件中约束最强、最推荐复用的默认模板
- 对话页、商业页、工作台页只应套用其所属类型规则与页面规则
- `AI Assistant` 与 `Billing` 不属于工具页，不应被强行套用“配置页 / 任务页”模板

## 3. 工具页核心原则

所有任务型工具都必须具备统一的过程意识，而不是一次性操作意识。

统一原则如下：

- 每个工具都必须有独立的“配置页”和“任务页”
- 任何真正消耗时间、Credit、积分、额度或会写入数据的操作，都必须先创建任务
- 创建任务前必须进行二次确认
- 任务页必须同时承载当前任务和历史任务
- 每个任务无论成功、失败、暂停或中断，都必须有结果产物或过程结果可查看
- 结果进入应用前，必须提供人工审核入口
- 所有任务都应尽量遵循 `创建 -> 进行 -> 审核 -> 应用` 的统一生命周期

### 3.1 不应套用工具页模板的页面

以下页面不应直接套用工具页模板，而应回到其所属的页面类型规则：

- AI Assistant：属于对话页
- Billing：属于商业页
- Workspace：属于工作台页

只有当某个页面的主用户心智明确是“发起一个任务、等待执行、审核结果、应用结果”时，才应进入工具页模板。

## 3.2 术语规范

所有面向用户、商家、运营人员的计费与消耗文案，统一使用 `Credit` 概念。

具体约束如下：

- 英文界面统一使用 `Credit / Credits`
- 中文界面统一使用“积分”或“额度”，表达同一套 `Credit` 语义
- 不应在同一用户界面中混用 `Token`、`Credit`、积分等多套计费单位叫法
- `Token` 仅保留给技术语境，例如 API token、access token、model token、design token

因此，任务消耗、套餐余额、按量购包、试用额度、历史流水等所有用户可见内容，都应按 `Credit / 积分` 命名。

## 4. 全局交互规则

## 4.1 页面层级与导航模型

Spark 中所有页面默认遵循以下层级：

- 应用级导航：用于切换核心工作域，例如 AI Assistant、工具页、Billing、Workspace
- 页面级模式切换：用于同一页面内部的主工作模式切换，例如 `配置页 / 任务页`
- 视图级切换：用于当前模式下的列表过滤、当前/历史切换、轻量分组切换
- 详情返回：用于从二级详情页回到来源列表或来源页面

约束如下：

- 应用级导航不得承担页内模式切换职责
- 页面级模式切换不得伪装成应用级导航
- 详情返回默认优先于重复展示主导航与主 tab
- 同一屏内不得同时出现三个层级都在竞争主视觉注意力

## 4.2 弹窗、抽屉与子页面决策

`Modal`、`Drawer`、`Detail Page` 的默认分工如下：

- 弹窗：用于确认、短说明、轻预览、单一动作补充
- 抽屉：用于轻量编辑、补充说明、辅助配置，不改变主工作区上下文
- 子页面：用于审核、对比、长编辑、批量处理、需要持续停留的工作空间

具体规则如下：

- 只要用户完成操作前需要持续阅读、对比或编辑，默认进入子页面
- 只要主工作目标依赖多个区块协作，默认进入子页面
- 只要关闭后会明显丢失工作上下文或中断判断，默认进入子页面
- 只有在“看完即可决策、只有一个主 CTA、关闭后不会损失主要工作状态”时，才适合使用弹窗

## 4.3 关闭语义规则

为了避免关闭方式语义漂移，默认遵循：

- 信息型弹窗：允许右上角 `X`
- 确认型弹窗：必须提供明确的 `取消 / 确认`，右上角 `X` 不得替代取消语义
- 工作流型弹窗：原则上不推荐；如果短期保留，主关闭动作必须在 footer 中明确出现
- 抽屉：默认应有清晰的关闭入口，且关闭不应伪装为主下一步动作

## 4.4 多级 tab 规则

同一页面最多允许两层可见 tab：

- 第一层：页面级模式切换
- 第二层：当前模式下的视图切换

约束如下：

- 第三层不得继续使用 tab；应改为筛选器、分段按钮、锚点或进入子页面
- 同层级 tab 必须使用一致的组件与一致的状态来源
- 工具内部 tab 不得混入应用级导航数据
- 仅因页内 tab 变化，不应触发高成本路由跳转或整页重载

## 4.5 主次动作层级

所有页面类型都应遵循：

- 同一主工作区只应有一个明确的主 CTA
- 次动作应弱于主 CTA，不得在视觉上与主 CTA 等权竞争
- 危险动作默认降级为次动作或收纳动作
- 辅助编辑、补充说明、继续优化等动作默认不得与“确认 / 应用 / 购买”并列为同级主动作

## 5. 工具页标准页面结构

每个工具默认由两个一级结构组成：

## 5.1 配置页

配置页用于发起任务。

配置页必须包含：

- 对应功能的参数配置表单
- 明确的主功能按钮
- 必要的辅助说明
- 任务创建前的必要提醒信息

配置页的职责是：

- 让用户明确这次任务要处理什么
- 让用户调整任务参数
- 触发任务创建

配置页不承担：

- 长时间执行过程展示
- 复杂历史任务承载
- 结果审核主承载
- 主操作后的成本确认主承载

## 5.2 任务页

任务页用于承载任务执行、任务结果、任务审核与历史记录。

任务页必须包含两个 tab：

- 当前任务
- 历史任务

对于同时包含“配置页”和“任务页”的工具页，工具内部默认还必须有一个一级页面切换：

- 配置页
- 任务页

该切换用于区分“创建任务”和“查看任务”的两种主工作模式。

约束如下：

- 该切换属于 tool 页面内部导航，不属于应用级主导航
- 该切换应作为页面标题下的第一层操作出现
- 该切换的数据源必须来自当前 tool 页自身的页面模式，而不是应用目录或功能目录
- `任务页` 可显示运行中任务数量徽标，但该徽标只用于提示，不应替代状态列表
- 该切换默认应以本地页面状态驱动，切换反馈必须接近即时，不应表现为整页重新加载
- 如果需要把当前页签同步到 URL，默认应使用轻量的地址同步方式，不应把页内 tab 切换实现为完整路由跳转
- 仅因为 `配置页 / 任务页`、`当前任务 / 历史任务` 等页内 tab 变化，不应重新触发鉴权、计费、任务列表拉取或其他重型 loader

### 5.2.1 页内 tab 性能规范

页内 tab 的目标是帮助用户在同一工具上下文中快速切换工作模式，而不是制造新的页面访问成本。

因此必须满足：

- `配置页 / 任务页` 切换应视为同页内模式切换，而不是新的一级路由导航
- `当前任务 / 历史任务` 切换应视为任务列表内部过滤切换，而不是新的数据页面
- 点击 tab 后，内容区应立即切换，不应出现明显等待、白屏、整页闪动或重新挂载的感知
- 如果业务需要保留可分享 URL，可以同步查询参数，但不得让该同步影响切换流畅度
- 页内 tab 不应因为地址同步而触发整个工具页的 loader、revalidate 或 embedded app 导航链路
- 页内 tab 的实现优先级应为：本地状态切换 > 轻量 URL 同步 > 完整路由导航

历史任务的定义：

- 创建时间超过 `24 小时` 的任务

任务页的职责是：

- 查看任务执行情况
- 查看任务流式进展
- 执行下一步动作
- 进入审核
- 查看历史结果

## 6. 任务创建与二次确认

点击配置页主功能按钮时，不应直接开始最终执行，而应先完成任务创建确认。

标准流程如下：

1. 用户在配置页填写参数
2. 用户点击主功能按钮
3. 系统弹出二次确认弹窗
4. 用户确认后创建任务
5. 系统跳转到任务页，并新增该任务

## 6.1 二次确认弹窗要求

二次确认弹窗必须向用户说明本次任务的基础情况。

至少应包含：

- 任务目标
- 核心参数摘要
- 预计影响范围
- 预估耗时
- 预估 Credit、积分或额度消耗
- 明确的确认按钮和取消按钮

对于会消耗时间、Credit、积分、额度或会触发真实写入风险的任务，预估耗时、预估消耗、影响范围等“执行前预估信息”默认只能承载在主操作后的二次确认弹窗中，不应在配置页中作为常驻信息模块重复出现。

配置页可以有轻量提醒文案，但这些提醒文案不能替代二次确认弹窗，也不应形成独立的常驻“执行前预估”卡片、侧栏或面板。

二次确认弹窗的目标是：

- 让用户理解这次任务会做什么
- 让用户理解这次任务要消耗什么
- 避免误触发高成本任务

## 7. 任务页结构

任务页必须以任务列表为核心承载。

每个任务项都应使用统一的任务卡片或任务行结构展示。

每个任务卡片至少需要展示以下信息：

- 任务 ID
- 任务目标
- 任务数量或处理对象数量
- 创建时间
- 更新时间
- 当前状态
- 已完成数量 / 总任务数量
- 百分比进度条
- 流式输出的进展说明
- 结果查看入口
- 审核入口
- 应用入口或后续操作入口

除上述基础字段外，任务卡片建议补充一行“任务详情摘要”，优先承载用户理解任务所需的上下文，而不是系统内部字段。

推荐摘要内容包括：

- 任务数量
- 输出语言
- 源语言
- 风格或品牌约束
- 处理对象标题或范围摘要

不推荐在摘要区长期承载：

- 内部资源 ID
- 技术性配置键
- 重复的计费实现字段
- 只有系统调试才需要理解的信息

## 7.1 当前任务

当前任务页签用于承载最近 24 小时内、仍在关注窗口内的任务。

其中通常包含：

- 正在创建的任务
- 正在进行的任务
- 待审核的任务
- 待应用的任务
- 最近完成的任务
- 最近失败或暂停但仍需处理的任务

## 7.2 历史任务

历史任务页签用于承载超过 24 小时的任务记录。

历史任务仍应保留：

- 任务基础信息
- 最终状态
- 结果查看入口
- 审核记录或应用记录

历史任务的重点不是继续操作，而是：

- 回溯
- 查询
- 查看结果
- 查看处理记录

## 8. 任务状态与动作

任务状态必须是显式的，不能让用户靠猜测理解任务进展。

工具可以按业务差异扩展状态，但至少应覆盖以下几类：

- 已创建
- 进行中
- 已暂停
- 待审核
- 已审核
- 待应用
- 已完成
- 失败

如果业务中存在“开始”与“创建”分离，也允许保留：

- 未开始
- 已开始

## 8.1 状态驱动动作

不同状态应提供不同的下一步按钮与操作按钮。

例如：

- 未开始 / 已创建：开始
- 进行中：查看进展、暂停
- 已暂停：继续、取消
- 待审核：去审核
- 已审核 / 待应用：去应用
- 已完成：查看结果
- 失败：查看原因、重试

动作设计原则：

- 用户永远能看到当前状态下最合理的下一步
- 不应让所有按钮长期同时出现
- 次要操作应弱于主下一步操作

当任务进入结果审核或应用阶段时，还必须满足以下优先级规则：

- 页面中只应有一个明确的主下一步动作
- 如果当前阶段的最终目标是“应用结果”，则“应用”必须是唯一主按钮
- 评分、补充说明、继续 AI 优化等动作默认视为辅助编辑动作，不应与主下一步动作抢占同一级视觉权重
- 辅助编辑动作可以使用次按钮、工具栏按钮或二级入口承载

## 9. 进度展示规范

任务页必须展示任务进度，且进度应同时包含数值进度和语义进度。

## 9.1 数值进度

数值进度至少包括：

- 已完成数 / 总任务数
- 百分比进度条

例如：

- `2 / 4`
- `50%`

## 9.2 语义进度

语义进度用于说明系统当前在做什么。

例如：

- 正在加载数据（2/4）
- 正在翻译
- 正在生成标题
- 正在写入结果
- 正在等待人工审核

语义进度应支持流式更新。

这类信息应尽量：

- 短
- 可扫读
- 与当前阶段一致

## 10. 结果与审核

每个任务不管最终状态如何，都应尽量产出对应的结果、阶段性结果或失败结果说明。

也就是说，任务不应只有“状态”，没有“产物”。

结果承载至少要支持：

- 查看当前结果
- 查看历史结果
- 对比前后内容
- 查看失败信息或中间产物

## 10.1 审核入口

所有任务都必须有结果审核入口。

审核入口可以采用：

- 弹窗
- 抽屉
- 独立详情页

具体形式可以按工具复杂度决定，但必须满足人工可审查。

复杂度判断默认遵循全局规则：

- 轻量确认与轻量预览可使用弹窗
- 辅助编辑与局部补充可使用抽屉
- 长对比、长编辑、审核与应用复合工作区默认进入独立详情页

如果审核或结果查看采用独立详情页承载，则该详情页默认视为任务页内部的二级页面，而不是新的一级导航页面。

这类二级详情页应遵循以下约束：

- 默认不再展示任务页的主导航、主 tab 或顶部统计模块
- 默认不再重复展示工具级主导航
- 页面顶部只保留返回上一页入口，用于回到任务列表
- 详情页主体应专注承载原文、结果、审核、评分、AI 优化与应用动作

如果详情页承担“审核 + 应用”复合任务，还应进一步遵循：

- 详情页头部使用一个统一的任务摘要区，而不是拆成多个分散的小头部
- 摘要区只放任务理解所需的关键信息，不放低价值系统字段
- 原文与结果对比区应成为页面主工作区，优先占据可视空间
- 评分、补充说明、继续 AI 优化等辅助编辑动作，默认使用弹窗、抽屉或局部面板承载
- 不应把多个可编辑表单长期平铺在主工作区下方，压缩内容对比空间

## 10.2 审核内容

审核区应支持以下一种或多种能力：

- 前后内容对比
- 原文与结果对比
- 差异高亮
- AI 质量评分
- 人工评分
- 人工编辑修正
- 补充说明与风险提示

审核区的目标不是“再展示一次结果”，而是支持人做最终判断。

对于内容生成类任务，推荐把“审核区”拆成以下关系：

- 主工作区：原文与当前结果并排对比
- 辅助编辑入口：结果评分、说明、继续 AI 优化
- 主下一步动作：应用当前结果

其中：

- 结果评分比“审核备注”更适合作为统一概念
- 评分说明属于评分的附属信息，不建议单独升格为独立主区块
- 继续 AI 优化属于基于当前草稿的迭代动作，应视为编辑行为，而不是并列主阶段

## 11. 应用阶段

人工审核确认后，任务应进入应用数据阶段。

应用阶段示例：

- 写入 Shopify
- 覆盖原字段
- 写入 metafield
- 推送到外部系统

应用阶段必须满足：

- 由明确动作触发
- 有单独状态反馈
- 有成功或失败结果

当页面同时存在“结果评分”“继续 AI 优化”“应用”三类动作时：

- “应用”必须在主界面中作为唯一主 CTA 暴露
- “结果评分”和“继续 AI 优化”应优先收纳为辅助入口
- 页面不应出现多个视觉上同等级的大按钮共同竞争用户注意力

如果应用会产生真实数据写入，就不应在任务执行完成时自动隐式发生。

## 12. 统一生命周期

所有任务型工具都应尽量遵循以下统一生命周期：

1. 创建
2. 进行
3. 审核
4. 应用

如果某些工具存在暂停、失败、重试等分支，也应被视为该生命周期的扩展状态，而不是独立体系。

## 13. 页面级信息层级

为了保持工具页一致性，页面应遵循以下信息层级：

### 配置页

- 页面标题与简要说明
- 参数配置表单
- 主功能按钮
- 任务创建确认入口
- 不承载常驻的执行前预估面板

### 任务页

- 当前任务 / 历史任务切换
- 任务列表
- 单任务进度与状态
- 结果入口
- 审核入口
- 应用入口

### 页面底部辅助区

- tool 页面允许在最底部放置低优先级辅助信息，例如语言切换、联系方式、支持邮箱或轻量帮助入口
- 该区域默认属于页面正文流的一部分，应在用户滚动到页面底部时自然露出
- 该区域不应默认做成悬浮条、吸底层或覆盖主内容的浮层
- 该区域不应与主操作区、任务卡、审核区竞争视觉层级
- 该区域应使用单行或紧凑布局，优先居中展示，避免扩展成新的信息卡片或营销模块
- 如果页面存在全局浮层、对话框、抽屉或详情弹窗，页面底部辅助区的层级必须低于这些交互层

## 14. 设计约束

为了防止后续实现偏离，本规范额外约束如下：

- 不允许把异步工具继续做成“一次点击立即写入”的黑盒交互
- 不允许缺失二次确认
- 不允许把执行前预估做成配置页常驻主模块，并与二次确认重复承载
- 不允许只有执行过程、没有审核入口
- 不允许只有审核结果、没有应用阶段
- 不允许历史任务彻底丢失结果查看能力
- 不允许不同工具完全发明不同的任务结构
- 不允许把页内 tab 切换实现成高成本路由跳转，并造成明显卡顿
- 不允许把页面底部辅助信息做成默认覆盖内容的悬浮层
- 不允许在已有共享交互基座时继续发明同类壳层而不说明理由

## 15. 与视觉规范的边界

本文件解决的是：

- 页面模式
- 流程结构
- 状态定义
- 交互阶段
- 审核与应用关系

本文件不解决：

- 卡片长什么样
- badge 用什么颜色
- tab 用什么视觉样式
- 使用何种阴影、圆角、字重和字号

## 16. 默认落地模板

未来所有新的任务型工具，在没有特殊说明时，默认采用以下模板：

1. 配置页填写参数
2. 点击主功能按钮
3. 弹出二次确认弹窗
4. 确认后创建任务并跳转任务页
5. 在当前任务中查看执行状态、进度与流式说明
6. 通过审核入口查看结果并进行人工审核
7. 审核通过后触发应用动作
8. 任务完成后进入任务记录，并支持后续查看结果

该模板应作为 Spark 工具页的默认交互骨架。

## 17. 商业页规则：Billing Page

`Billing Page` 不属于标准任务型工具页，但属于 Spark 的核心商业页，需要单独遵循商业页类型规则与 Billing 页面规则。

Billing 页的目标是：

- 让用户快速理解当前账户资源状态
- 让用户理解订阅计划与积分购买的差异
- 让用户完成订阅或加购
- 让用户查询积分历史变动与消耗记录
- 让用户理解不同计划的功能边界与常见问题

## 17.1 页面结构

Billing 页默认由以下几个核心区域组成：

1. 账户概览区
2. 订阅计划区
3. 积分购买区
4. FAQ 区

页面可以按业务需要调整顺序，但默认建议保持以上阅读顺序。

## 17.2 账户概览区

头部必须有独立的账户区域，用于展示当前账户资源状态。

至少需要展示：

- 当前积分余额
- 当前订阅状态
- 当前生效计划
- 计划周期信息
- 下次续费或到期时间
- 积分明细入口

如果账户信息较多，首页默认应只承担：

- 资源状态概览
- 账户详情入口

账务流水、周期归档、订阅明细等信息较多的内容，默认应收拢到 Billing 内部二级页面，而不是在主页面中并列堆叠多个大卡片。

积分明细应支持查看历史增减记录。

积分明细至少需要覆盖：

- 订阅发放
- 积分购买
- 任务消耗
- 手动调整或补偿
- 退款或回退

如果信息较多，可以使用：

- 明细弹窗
- 抽屉
- 独立明细页

但入口必须在头部账户区域中明确可见。

如果采用独立账户详情页承载，则该页面默认应遵循：

- 作为 Billing 页内部二级页面存在
- 顶部只保留返回上一页入口
- 主体按纵向信息流展示订阅、账务流水、周期归档
- 不重复展示套餐选择、购包和 FAQ 等主页面模块

## 17.3 订阅计划区

Billing 页必须支持“订阅计划”和“积分购买”两种商业模式并存。

订阅计划区用于展示不同付费计划。

至少需要支持：

- 月付
- 年付

其中年付必须展示相对月付的折扣信息。

订阅计划区至少应说明：

- 每个计划的价格
- 计费周期
- 核心差异
- 包含的基础额度或权益
- 当前计划标记
- 升级、降级、订阅、续费等入口

如果当前用户已经订阅某计划，也应能清楚看到：

- 当前所在计划
- 可升级到哪些计划
- 切换计划的结果或影响

## 17.4 月付与年付切换

月付与年付应有统一切换控件，不应让用户在多个分散模块中自行比对。

切换后至少要同步更新：

- 价格展示
- 折扣展示
- 计划卡 CTA 文案
- 相关说明文字

折扣信息必须清晰，但不能掩盖计划本身的权益差异。

## 17.5 积分购买区

Billing 页必须提供积分购买区域，作为订阅之外的补充消费能力。

积分购买区至少需要展示：

- 可购买积分包
- 对应价格
- 获得积分数量
- 购买入口

积分购买和订阅计划必须被明确区分，避免用户误以为它们是同一种计费方式。

需要明确说明：

- 订阅适合什么场景
- 积分购买适合什么场景
- 两者是否可以同时存在

## 17.6 计划差异说明

Billing 页需要说明计划差异，但不一定要单独再放一个与套餐选择重复的大模块。

默认要求是：

- 计划差异说明优先并入订阅计划主决策区
- `Choose the plan that fits you` 这一主区域应同时承担价格、周期、核心差异和 CTA
- 不建议在其后再次追加一个语义重复的 `Plan details` 独立大模块

如果确实需要补充说明，也应只补充套餐卡中难以承载的差异信息，而不是重复已有价格、卖点和 CTA。

## 17.7 FAQ 区

Billing 页底部必须有 FAQ 模块，用于解答付费相关的常见问题。

FAQ 至少建议覆盖：

- 订阅如何计费
- 月付和年付有什么区别
- 年付折扣如何计算
- 如何购买积分
- 积分是否会过期
- 任务消耗积分的规则
- 升级或降级计划后会发生什么
- 取消订阅后会发生什么
- 如何查看积分明细与历史消耗

FAQ 应作为页面底部的长期帮助区，整个 FAQ 模块必须默认可见；单个问答项可以折叠，但不得把整个 FAQ 区域默认收起到难以发现的位置。

## 17.8 账户与消费透明度

Billing 页必须强调账户状态和消耗透明度。

因此必须满足：

- 用户能看到自己当前拥有什么资源
- 用户能看到资源如何被消耗
- 用户能看到历史增减记录
- 用户能理解计划与积分的关系
- 用户能理解每次付费动作的结果

如果某个操作会改变订阅或积分状态，也应提供明确的确认或结果反馈。

## 17.9 与任务型工具页的区别

Billing 页不是“配置页 + 任务页”结构，也不强制遵循“创建 -> 进行 -> 审核 -> 应用”的任务链路。它属于商业页，而不是工具页例外。

Billing 页的默认模式是：

- 账户状态查看
- 商业方案选择
- 购买或订阅决策
- 历史明细查询
- 常见问题解答

因此 Billing 页应被视为 Spark 中的商业与账户页例外模式，而不是任务型工具模板的一部分。

## 17.10 Billing Page 默认信息架构

如果没有特殊说明，Billing 页默认采用以下信息架构：

1. 页面标题与说明
2. 资源状态概览区
3. 账户详情入口
4. 订阅计划区
5. 积分购买区
6. FAQ 区

## 17.10.1 页面标题与说明

页面顶部应先回答用户当前进入该页的目的。

建议包含：

- 页面标题
- 一句简短说明
- 如有必要，展示当前额度不足、订阅待确认或资源告警

## 17.10.2 账户概览区

账户概览区是 Billing 页的第一优先级信息区。

建议拆成两个层次：

- 资源状态概览
- 明细与历史入口

其中资源状态概览应至少包含：

- 当前可用积分
- 已消耗积分
- 订阅状态
- 当前计划
- 周期结束时间或到期时间

明细与历史入口默认应能进入账户详情二级页，承接订阅明细、账务流水和周期归档等更完整的信息。

## 17.10.3 订阅计划区

订阅计划区是 Billing 页的核心决策区。

默认结构建议为：

- 月付 / 年付切换
- 计划卡列表
- 当前计划标识
- 推荐计划标识
- 订阅或变更计划按钮

计划卡中至少需要展示：

- 计划名称
- 价格
- 周期
- 年付折扣
- 核心额度
- 核心卖点
- 当前状态

## 17.10.4 积分购买区

积分购买区应用于补充型消费，不应与订阅计划混在同一组卡片中。

默认结构建议为：

- 积分包选项
- 当前选中积分包摘要
- 购买按钮
- 与订阅的关系说明

如果同时存在订阅和积分购买，页面必须说明：

- 订阅提供什么
- 积分包补充什么
- 哪类用户适合买包而不是升级订阅

## 17.10.5 计划差异说明

计划差异说明默认并入计划选择主区，不再要求单列一个独立的大型详解模块。

至少应覆盖：

- 价格与周期差异
- 核心额度差异
- 当前计划与推荐计划状态
- 用户应选择某个计划的主要理由

## 17.10.6 FAQ 区

FAQ 区默认置于页面底部。

建议采用折叠列表或问答块结构。

每个问题应能够独立阅读，不依赖前文上下文。

## 17.10.7 积分明细承载方式

Billing 页必须提供积分历史增减明细的完整承载方案。

默认可选方案包括：

- 在账户概览区提供“查看明细”入口，打开弹窗
- 在账户概览区提供“查看明细”入口，打开抽屉
- 在 Billing 页中提供独立明细区块
- 跳转独立明细页

积分明细至少应包含：

- 记录时间
- 变动类型
- 增减值
- 变动后余额
- 原因说明
- 关联任务或订单信息

如果是任务消耗，应尽量能看到：

- 任务来源
- 工具名称
- 消耗数量
- 对应任务或结果入口
